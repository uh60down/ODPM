# ODPM at a Glance

**Ontology-Driven Project Management (ODPM)**

> ODPM is a framework for building a shared ontology before execution.

It does not replace existing project management frameworks.
It complements them by making the team's shared model of the world explicit before planning, implementation, and execution begin.

---

## The Framework

```text
Intent
(PRD · Brief · Charter · Mandate)

        ↓

Ontology
(Shared Model of the World)

        ↓

Decision Readiness

        ↓

Execution

        ↓

Feedback

        ↓

Ontology Evolves
```

Execution tests the ontology.
Feedback refines it.

---

## The Building Blocks of an Ontology

An ontology is built from four fundamental elements.

### Concepts

The things that exist.

Examples:

- User
- Vehicle
- Campaign
- OTA Update

Ask:

> What exists?

---

### Relationships

How concepts are connected.

Examples:

- User owns Vehicle
- Vehicle receives OTA Update
- Campaign targets Vehicle

Ask:

> How are they related?

---

### Rules

The constraints that govern the world.

Examples:

- Only Owners can start an OTA Update.
- Scheduling must be within 24 hours.

Ask:

> What must always be true?

---

### States

The current reality of a concept.

Examples:

Available → Downloading → Installing → Completed

Ask:

> What is true right now?
>
> What is possible next?

## Core Principles

- A shared ontology before execution.
- Questions over assumptions.
- Clarity over complexity.
- Principles over procedures.
- Alignment over individual brilliance.
- Decision Readiness, not complete understanding.
- Every artifact is a snapshot. Ontology evolves.

## Glossary

**Ontology**

The team's shared model of the world.

---

**Concept**

Something that exists in the world the team is building.

---

**Chase Understanding**

Using questions to discover reality together.

---

**Decision Readiness**

The team's current ontology is sufficient to take the next responsible action.

---

**Snapshot**

The team's current best expression of the ontology at a point in time.

---

**Source Artifact**

A PRD, brief, charter, mandate, or other artifact containing intent, requirements, assumptions, and constraints from which an ontology can be discovered.

## Where ODPM Lives

Every framework has a moment where understanding becomes execution.

ODPM lives immediately before that moment.

| Framework | Execution Begins | ODPM Position |
|-----------|------------------|---------------|
| GitHub | Commit / Merge | Before the ontology becomes code |
| Scrum | Sprint | Before the ontology becomes work |
| PMBOK | Planning & Execution | Before the ontology becomes a plan |

ODPM is framework-agnostic because it sits upstream of execution, regardless of the framework being used.
